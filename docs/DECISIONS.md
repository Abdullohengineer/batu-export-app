## 2026-07-12 ÔÇö Phase 0 database migrations
**Context:** Turning SPEC ┬º8 + PHASE0 Part B into numbered SQL migrations in `/supabase/migrations`.
**Decision:**
- Followed PHASE0's file grouping but numbered/split it as 7 files so master data always precedes anything that references it: `0001_master_data`, `0002_serial`, `0003_operational`, `0004_storage`, `0005_notes_audit`, `0006_views`, `0007_rls`.
- `next_serial()` is `security definer` (search_path pinned to `public`) so it can write to `serial_counter` even though that table itself carries **no RLS policies at all** ÔÇö the function is the only door in or out of it. Matches "the database is the source of truth" (┬º2.15) without opening a side channel.
- `my_role()` is also `security definer` so it can read `profiles` regardless of the caller's own row-level access.
- RLS is enabled on all 20 tables that exist after these migrations (not just the sample list PHASE0 spelled out), including `serial_counter`, `product_categories`, `product_types`, `calibres`, `settings_limits`, and `profiles`. Verified locally: with RLS on, a `rahbar`-role user is refused an `insert` into `kirim_orders`, and a `menejer`-role user succeeds ÔÇö the exact test PHASE0 Part F asks for.
- `audit_log` SELECT is restricted to `rahbar` (not "any authenticated user") since it's raw compliance/oversight data with no operational screen reading it directly; every other operational table keeps the broad "any authenticated user can read" policy PHASE0 used, since Kuzatuv/traceability need cross-role visibility.
- Added `created_by uuid references profiles` to `gate_weighings` for consistency with the other operational tables (`kirim_orders`, `chiqim_requests`, etc. all record who created the row) ÔÇö PHASE0's sample SQL omitted it, likely an oversight.
- SPEC ┬º8's `USER_PREFS` (user_id, language) was **not** created as a separate table ÔÇö `profiles.language` (from PHASE0's B1) already satisfies ┬º2.8's per-user Uz/Ru toggle, and a second table would just be a duplicate write target.
- Smoke-tested all 7 migrations against a real local Postgres 16 instance (not just reviewed): they apply cleanly in order, `next_serial()` returns `DDMMYY-NNN` using Tashkent's date, `net_kg` computes as a generated column and can't be overridden by an insert, and the RLS refusal/success behavior above was verified with actual role-switching.

## 2026-07-12 ÔÇö Auth method: phone number, not email
**Context:** BATU has no SMS provider budget and no real inboxes for staff logins ÔÇö an email-based flow (verification links, password reset) doesn't work for a warehouse/gate/lab workforce logging in by phone.
**Decision:**
- Login screen collects **phone number + password**. Under the hood this is still Supabase email/password auth: the phone is converted to a synthetic address `<digits>@batu.local` (see `src/lib/phoneAuth.ts`, shared by the login screen and the admin function) before calling `signInWithPassword` / `createUser`. No SMS provider, no per-login cost.
- **Consequence:** Supabase's built-in "forgot password" (magic link to a real inbox) is unusable ÔÇö there is no inbox behind `@batu.local`. No self-service signup or forgot-password UI was built; public signup stays disabled (Supabase Auth setting, outside this repo). The login screen shows a static "contact Rahbar" hint instead.
- Replacement: a Supabase Edge Function (`supabase/functions/admin-users`) using the `service_role` key, is the only way to create a user or reset a password. It checks the caller's own JWT, looks up their `profiles.role`, and returns 403 unless it is `rahbar` and `active` ÔÇö enforced in the function itself, not just by hiding the "Foydalanuvchilar" screen (┬º2.12's server-side-enforcement principle, applied to admin actions too).
  - `create_user`: `{ phone, password, role, full_name }` ÔåÆ `auth.admin.createUser` + matching `profiles` insert (with the new `profiles.phone` column added in `0008_profiles_phone.sql`, used to look users up by the number they actually log in with). If the `profiles` insert fails, the orphaned auth user is deleted so a retry with the same phone doesn't collide ÔÇö this is an admin API cleanup of Supabase's own `auth.users`, not a `DELETE` against any of our operational tables, so it doesn't break the "never DELETE, only void" house rule (┬º2.15).
  - `reset_password`: `{ phone, new_password }` ÔåÆ looks up the id by `profiles.phone`, then `auth.admin.updateUserById`.
- Added a **"Foydalanuvchilar"** nav item + screen (`src/pages/admin/UsersAdminPage.tsx`), visible only when `profile.role === 'rahbar'`, with the two forms above calling the Edge Function via `supabase.functions.invoke`.
- This is the first role-gated screen, so `App.tsx`/`HomePage.tsx` now fetch the caller's `profiles` row (`useProfile` hook) after login. Still no router ÔÇö the rahbar's home/users toggle is local `useState`, same reasoning as the earlier "no router yet" decision. Full per-role home screens and a route guard are still deferred to the dedicated auth/roles step.

## 2026-07-13 ÔÇö Edge Function secret renamed to SERVICE_ROLE_KEY
**Context:** `npx supabase secrets set SUPABASE_SERVICE_ROLE_KEY=...` was rejected ÔÇö the Supabase CLI reserves the `SUPABASE_` prefix for its own auto-injected environment variables and refuses to let you set a custom secret under that prefix.
**Decision:** The service-role secret for `admin-users` is set and read as **`SERVICE_ROLE_KEY`** (no `SUPABASE_` prefix), not `SUPABASE_SERVICE_ROLE_KEY`. Updated `supabase/functions/admin-users/index.ts` accordingly (`Deno.env.get('SERVICE_ROLE_KEY')`). `SUPABASE_URL` is unaffected ÔÇö that one is one of the platform's own auto-injected vars, not a user-set secret, so it keeps its name.
**Action required outside this repo:** set the secret and redeploy ÔÇö `npx supabase secrets set SERVICE_ROLE_KEY=<your service_role key>` then `npx supabase functions deploy admin-users`. This sandbox cannot reach `api.supabase.com` (network policy blocks it), so both commands need to be run from a machine with real network access to Supabase.

## 2026-07-13 ÔÇö CORS headers on admin-users
**Context:** The browser was blocking the call from the Foydalanuvchilar screen ÔÇö `supabase.functions.invoke` sends a CORS preflight (`OPTIONS`) before the real `POST`, and the function had no CORS handling at all, so the preflight got a plain 404/405 with no `Access-Control-Allow-Origin` and the browser refused to even attempt the POST.
**Decision:**
- Added a shared `CORS_HEADERS` object (`Access-Control-Allow-Origin: *`, `Access-Control-Allow-Headers: authorization, content-type, apikey, x-client-info`, `Access-Control-Allow-Methods: POST, OPTIONS`) and an explicit `OPTIONS` branch at the top of `Deno.serve` returning `204` with just those headers, no body.
- Routed **every** other response ÔÇö success and every error path (401 missing/invalid session, 403 non-rahbar, 400 validation, 404 not found, 405 wrong method) ÔÇö through the existing `json()` helper, and added `...CORS_HEADERS` to the headers it sets. Because every `return` in the function goes through `json()` (or the new OPTIONS branch), there's no code path that can respond without CORS headers ÔÇö including the 403 rahbar-check rejection, which is exactly the kind of error response that's easy to forget when only the "happy path" gets CORS treatment.
- `apikey` and `x-client-info` are included in `Access-Control-Allow-Headers` alongside `authorization`/`content-type` because the supabase-js client sends those on every request by default; leaving them out would have caused the same preflight failure for a different header.
**Action required outside this repo:** redeploy ÔÇö `npx supabase functions deploy admin-users`. Same network caveat as above: this sandbox can't reach Supabase to do it directly.

## 2026-07-13 ÔÇö Temporary diagnostic markers (OPTIONS still 405ing after redeploy + --no-verify-jwt)
**Context:** After redeploying with `--no-verify-jwt`, `curl -X OPTIONS` against the function still returned the function's own `{"error":"Method not allowed"}` (the 405 body from `json()`), even though `if (req.method === 'OPTIONS')` is the literal first statement in `Deno.serve` ÔÇö there is nothing in this file that runs before it and could reject the request first. That points at the deployed bundle not matching this file (wrong linked project, a deploy that silently didn't take, etc.) rather than a logic bug in the handler.
**Decision:** Added temporary, clearly-marked diagnostic code (search `TEMPORARY DIAGNOSTIC` to find/remove it later):
- A `DIAG_VERSION` string logged once at module load (`console.log`, so it shows up in `supabase functions logs admin-users` even for requests that never reach the handler at all).
- The same marker echoed as an `X-Admin-Users-Diag` response header on **every** response (added to `CORS_HEADERS`, which both `json()` and the OPTIONS branch already use) ÔÇö so `curl -i` shows it directly without needing log access.
- A `console.log` as the literal first line inside `Deno.serve`, before the OPTIONS check, logging method + URL + the version string.
If `curl -i -X OPTIONS <url>` still comes back without `X-Admin-Users-Diag` after redeploying this version, the request is not reaching this function's code at all ÔÇö the problem is upstream (project link, deploy target, routing), not in this file. If the header *is* present but still 405, the bug is genuinely inside this handler and needs a different fix. Remove this block once the real CORS behavior is confirmed working.

## 2026-07-14 ÔÇö Serial is per type-line, not per truck
**Problem:** a serial was defined as one-per-truck. A multi-type delivery (e.g. one truck carrying Subxon + Isfara) therefore shared a single Barcode #1, a single lab result, and a single `STORAGE_STOCK` row across two different products ÔÇö silently applying one type's moisture/SOÔéé reading to another, and making per-type traceability impossible. Three existing rules (Barcode #1 per type-pile, Laborator's one-sample-per-serial, `STORAGE_STOCK` keyed by serial) were only *approximately* true because of this ÔÇö they described what should happen per type, while the data model only tracked per truck.
**Decision:** mint one serial per type line, not one per truck. `next_serial()` is now called once per KIRIM type row, not once per KIRIM order. A serial is therefore single-type **by construction** ÔÇö the three rules above go from approximate to actually correct.
**Rejected alternative:** keep one serial per truck and apportion the gate's net weight pro-rata across the truck's types. Rejected because it invents a weight nobody measured ÔÇö and the storage manager physically separates the piles into their own stacks anyway when unloading, so the real per-type number is available for free instead of being estimated.
**What changes:**
- **Manager (┬º3.1):** a KIRIM order is now a *delivery envelope* (one truck-trip); each type row on it mints its own serial. The quantities on the form are the manager's **declared** figures ÔÇö never overwritten.
- **Storage (┬º5.1):** enters the **actual** measured weight per serial while unloading, and does so against a form that always shows the matching declared figure next to it (Seriya ┬À Tur ┬À Buyurtma (kutilgan) ┬À Aniq), with live variance and a "Kam chiqdi" flag ÔÇö so a shortfall is caught on the floor during unloading, while the pile is in front of him, rather than surfacing later in a report. Prints Barcode #1 per serial.
- **Laborator (┬º5.5):** "one sample ÔåÆ whole parent serial" is now genuinely correct, since the parent serial can no longer span more than one type.
**Does NOT change:** Qorovul's workflow. He still weighs one truck, twice ÔÇö one ðôÐÇÐâðÂÐæð¢Ðïð╣, one ðƒÐâÐüÐéð¥ð╣, one net ÔÇö regardless of how many types/serials are on it. Multi-serial trips are display-only at the gate.
**Consequence:** `GATE_WEIGHINGS.ref` now points at the trip (`order_id` for KIRIM / `request_id` for CHIQIM), never at a serial ÔÇö one weighing belongs to a truck-trip, not to any single type on it. The gate's net is reconciled against the manager's declared **Jami avto** (truck total vs. truck total), not split across serials. Declared (manager, ┬º3.1) and actual (storage, ┬º5.1) are separate persisted quantities; their gap is what "Kam chiqdi" means, and it is a finding to be recorded, never a number to be corrected away.

## 2026-07-14 ÔÇö Phase 1 Step 1: KIRIM_ORDERS/KIRIM_LINES migrated to v1.9; discovered a bigger FK ripple than scoped
**Context:** Building the Menejer's KIRIM tab against the v1.9 schema (previous entry). Phase 0's `kirim_orders`/`kirim_lines` still had the old one-serial-per-truck shape (serial lived on `kirim_orders`), with no application built against them and no real data ÔÇö so `0009_kirim_serial_per_type.sql` drops and recreates both tables in the new shape rather than migrating data.
**Discovered while migrating, not anticipated by the task scope:** the task said "GATE_WEIGHINGS changes ÔÇö not touched by this step," but the actual dependency graph is much wider. Six tables (`gate_weighings`, `storage_intake`, `moyka_sends`, `finished_pallets`, `wash_cycles`, `lab_results`) all carry a `serial` column with a foreign key to `kirim_orders(serial)`, and the `v_serial_balance` view is built entirely on `kirim_orders.serial` as its join anchor. Once `serial` moves off `kirim_orders`, none of that is satisfiable as written ÔÇö this isn't optional cleanup, the migration cannot run at all otherwise (Postgres refuses to drop a column that six FKs and a view depend on).
**Decision ÔÇö minimal fix only, most of it deliberately not done here:**
- Dropped `v_serial_balance` outright (its definition is invalid the moment `kirim_orders.serial` is gone) and dropped the six now-unsatisfiable FK *constraints*. The six tables keep their `serial` **column** ÔÇö just unconstrained for now ÔÇö nothing about their shape or behavior otherwise changed.
- Did **not** re-point those FKs at `kirim_lines(serial)` and did **not** rebuild the view. That's real design/implementation work belonging to Gate (┬º4), Storage ┬º5.1, Laborator (┬º5.5), and Kuzatuv (┬º3.4) respectively ÔÇö none of which are in scope for this step, per the task's explicit instructions and its own "GATE_WEIGHINGS not touched" carve-out extended to the same principle for the other five.
- **Flagging this loudly rather than silently fixing it**, matching the same policy used for the SPEC v1.9 doc scan: whoever builds those next steps needs to know `storage_intake.serial`, `moyka_sends.serial`, `finished_pallets.serial`, `wash_cycles.serial`, `lab_results.serial`, and `gate_weighings.serial` are currently **not** FK-enforced against anything, and `v_serial_balance` does not exist at all until it's rebuilt against `kirim_lines`.
**Also added, as necessary supporting infrastructure (not scope creep):** a `kirim-photos` Supabase Storage bucket + RLS policies (`0010_kirim_photos_bucket.sql`), since `doc_photo` is documented as "storage path" and the KIRIM form's Hujjat rasmi field is explicitly required for this step ÔÇö no bucket existed anywhere in the project yet. Same read-all/menejer-writes shape as every other RLS policy in this schema. Could not be verified locally (this sandbox's Postgres has no `storage` schema ÔÇö that only exists on an actual Supabase project), unlike every other change in this migration, which was verified end-to-end against a real local Postgres instance: all four SQL statements types (drop view, drop constraints, drop/recreate tables, RLS policies) applied cleanly in order; a two-type order produced two distinct serials referencing the same `order_id`; a `menejer` insert succeeded and a `qorovul` insert was refused with SQLSTATE `42501`, both against the new table shape.
**UI scope note:** `declared_qty`/status display in the live order list shows the *order's* status next to each line, not a per-line status ÔÇö `kirim_lines` in the required v1.9 schema has no status column of its own (status lives on `kirim_orders`); nothing in ┬º8 v1.9 suggested adding one, so nothing was added.

## 2026-07-14 ÔÇö Phase 1 Step 2: Qorovul KIRIM gate; gate_weighings finished migrating to v1.9
**Context:** Building Qorovul's KIRIM two-stage weighing tab. The task asked to add `order_id` to `gate_weighings` and treat `request_id` as "unconstrained for now (chiqim_requests doesn't exist yet)."
**Correction to the task's assumption, found by inspecting the live table first (as instructed) rather than trusting the prompt:** `chiqim_requests` already exists ÔÇö it was created in Phase 0's `0003_operational.sql` and never removed ÔÇö and `gate_weighings.request_id` already had a working FK to it. So `request_id` needed no change at all; only `order_id` was new. `net_kg` was also already a generated column, and RLS on `gate_weighings` already exactly matched the required pattern (read-all select, qorovul insert, qorovul update). The only real gap was the old `serial` column (unconstrained since the previous entry's migration) and its check constraint, which referenced a column that no longer means anything under v1.9.
**Decision (`0011_gate_weighings_order_id.sql`):** added `order_id` (FK ÔåÆ `kirim_orders`), replaced the old `serial`-based check constraint with `num_nonnulls(order_id, request_id) = 1` (a weighing belongs to exactly one trip, KIRIM or CHIQIM, never both, never neither), dropped `serial`.
**Status flip (`0013_gate_weighings_complete_kirim.sql`):** `kirim_orders.status` flips `kutilmoqda ÔåÆ qabul_qilindi` via a `security definer` trigger on `gate_weighings`, firing only when `completed_at` transitions from null to set on a `kirim`-direction row. This has to be a trigger, not a client-side update: `qorovul` has no write policy on `kirim_orders` at all (by design, per ┬º2.15 ÔÇö only `menejer` may insert there), so the app itself is structurally incapable of flipping this status from anywhere except stage-2 completion.
